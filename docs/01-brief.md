# Общее описание проекта: S3-клиент

## 1. Назначение проекта

Разработка десктопного кросс-платформенного приложения для работы с S3-совместимыми облачными хранилищами. Приложение решает проблему существующих клиентов, которые работают медленно, нестабильно на слабых соединениях и потребляют избыточные ресурсы.

## 2. Проблема, которую решаем

Существующие решения (AWS CLI, s3cmd, S3 Browser, Cyberduck, различные Electron-обёртки) страдают от системных недостатков:

- **Python-инструменты** (AWS CLI, s3cmd) ограничены GIL, показывают низкую скорость передачи и плохо масштабируются на многопоточные операции.
- **Electron-приложения** потребляют 300–600 МБ RAM и имеют латентность UI из-за тяжёлого runtime.
- **Существующие GUI-клиенты** плохо работают с не-AWS S3-совместимыми провайдерами (Yandex Cloud, Selectel, MinIO, Ceph), часто не поддерживают кастомные endpoint'ы и регионы.
- **Нестабильность на слабых каналах**: стандартные клиенты используют фиксированные таймауты и не умеют агрессивно ретраить зависшие соединения.
- **Неудобный UX**: отсутствие нормального drag-and-drop, прогресса передачи с оценкой времени, массовых операций.

## 3. Что строим

Десктопное приложение с нативной производительностью сетевого движка и современным веб-интерфейсом.

**Ключевые принципы:**
- **Скорость передачи** должна максимально приближаться к пропускной способности канала пользователя.
- **Стабильность** на нестабильных и медленных соединениях через умный retry и адаптивную стратегию передачи.
- **Лёгкость** приложения — запуск за секунды, потребление RAM < 150 МБ в простое.
- **Универсальность** — работа с любым S3-совместимым API (AWS, Yandex Cloud, MinIO, Ceph, DigitalOcean Spaces, Wasabi и другие).

## 4. Целевая аудитория

- Системные администраторы, работающие с S3-хранилищами ежедневно.
- Разработчики, загружающие артефакты, бэкапы, статику в облако.
- Контент-менеджеры и дизайнеры, работающие с медиа-библиотеками в облачных бакетах.
- Пользователи с ограниченным интернет-каналом, которым нужна надёжная докачка и устойчивость к обрывам.

## 5. Ключевые функции

### Управление подключениями
- Хранение нескольких профилей подключений к разным S3-совместимым хранилищам.
- Поддержка кастомных endpoint, регионов, путей стиля (path-style vs virtual-hosted-style).
- Проверка соединения и прав доступа при сохранении профиля.

### Файловый менеджер
- Двухпанельный или однопанельный интерфейс просмотра бакетов и объектов (как в Finder/Explorer).
- Навигация по префиксам (папкам), отображение размера, даты изменения, MIME-типа.
- Предпросмотр изображений, текстовых файлов, PDF непосредственно в интерфейсе.
- Поиск по объектам в бакете с фильтрацией.

### Передача файлов
- Загрузка файлов и папок в S3 через drag-and-drop или диалог выбора.
- Скачивание объектов и папок (рекурсивно) на локальный диск.
- **Параллельная передача** — многопоточная загрузка/скачивание с отображением скорости, прогресса и оставшегося времени.
- **Докачка** — возможность возобновить прерванную загрузку или скачивание.
- **Очередь передач** — постановка задач в очередь, приоритизация, пауза и отмена.

### Массовые операции
- Копирование, перемещение, удаление объектов внутри хранилища.
- Массовое удаление с подтверждением.
- Изменение метаданных объектов (Content-Type, кастомные теги).
- Генерация presigned URL для временного доступа к объекту.

### Безопасность и удобство
- Локальное хранение учётных данных (не в облаке, не в браузере).
- Поддержка IAM-ролей, STS-токенов, временных credentials.
- Тёмная и светлая тема интерфейса.
- Горячие клавиши для основных операций.

## 6. Технологический стек

### Backend / S3 Engine: Go
- **Причина выбора**: goroutines дают эффективный параллелизм без сложностей async/await, компиляция в нативный код обеспечивает высокую производительность сетевых операций, s5cmd доказал, что Go идеален для S3-задач.
- **Ключевые библиотеки**: стандартный `net/http` с пулом соединений, `aws-sdk-go-v2` для S3-протокола, `fsnotify` для мониторинга локальных папок (если потребуется синхронизация).

### Frontend / UI: Wails v2
- **Причина выбора**: фреймворк для создания десктопных приложений с Go-backend и веб-frontend. Использует нативный WebView ОС, что даёт лёгкие бинарники и низкое потребление памяти по сравнению с Electron.
- **Frontend-фреймворк**: React с TypeScript (или Vue 3 — на усмотрение разработчика, React предпочтительнее для экосистемы компонентов).

### Сборка и распространение
- **Кросс-компиляция**: Wails поддерживает сборку под Windows, macOS и Linux из одной кодовой базы.
- **Установщики**: `.exe` (Windows), `.dmg` (macOS), `.AppImage`/`.deb` (Linux).
- **Автообновление**: интеграция механизма автообновления через GitHub Releases (wails обеспечивает базовую инфраструктуру).

## 8. Отличительные особенности продукта

1. **Производительность на уровне CLI-инструментов** — скорость передачи сопоставима с s5cmd, но с удобством GUI.
2. **Устойчивость к сетевым проблемам** — приложение не падает при обрыве соединения, адаптирует стратегию передачи под реальную скорость канала.
3. **Универсальность S3-провайдеров** — одинаково хорошо работает с AWS, отечественными облаками и self-hosted решениями (MinIO).
4. **Лёгкость и скорость запуска** — запуск за 1–2 секунды, отсутствие тяжёлого runtime.
5. **Нативный UX** — интерфейс не отличается от привычных файловых менеджеров, поддержка drag-and-drop, горячих клавиш, системных диалогов.

## 9. Критерии качества и успеха

- Приложение загружается быстрее 2 секунд на среднем ноутбуке.
- Скорость загрузки/скачивания больших файлов (> 1 ГБ) не уступает s5cmd при аналогичных условиях.
- Приложение корректно обрабатывает обрыв соединения и возобновляет передачу.
- Потребление RAM в простое не превышает 150 МБ.
- Интерфейс остаётся отзывчивым во время активных передач.
- Поддержка всех основных операций с клавиатуры без необходимости использования мыши.

## 10. Этапы реализации

**Этап 1. Ядро и подключения** — базовый S3-движок на Go, управление профилями, простой файловый менеджер в UI.
**Этап 2. Передачи** — загрузка и скачивание файлов, очередь, прогресс, drag-and-drop.
**Этап 3. Массовые операции и оптимизации** — параллельный multipart, retry-логика, массовые операции, предпросмотр.
**Этап 4. Полировка** — тёмная тема, горячие клавиши, автообновление, инсталляторы, тестирование на всех платформах.
