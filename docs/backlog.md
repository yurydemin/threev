# Бэклог доработок

> Актуально на v0.4.3 (2026-07-24). Ничего из перечисленного не блокирует релизы — реализация по мере необходимости.

## Функциональность

Отсортировано по приоритету (высокий → низкий). Разделы «Высокий» (отдельный экран «История», избранное/закладки, CopyObject/MoveObject >5 ГБ, поиск по всему бакету, дашборд размера бакета, экспорт/импорт профилей, превью видео) и «Средний» (дерево папок в Copy/Move, управление бакетами, retry/timeout в UI, ZIP-скачивание) полностью реализованы в Этапах 6–7 (v0.3.1–v0.4.1) — см. [README.md](../README.md) и `git log`. HTTP/SOCKS5-прокси (per-connection) реализован в Этапе 8 Блок A (v0.4.3).

**Низкий**

- **Копирование между разными подключениями** — сейчас Copy/Move работает только внутри одного подключённого профиля; нет способа перенести объект между двумя разными аккаунтами/хранилищами напрямую. В работе — Этап 8 Блок B.
- **Предупреждение о закрытии при активной работе** — окно закрывается без предупреждения, даже если есть незавершённые передачи или открыта сессия подключения. В работе — Этап 8 Блок C.

## Технический долг

- `transfer_queue.parts_completed`/`file_offset` — колонки фактически не используются (источник истины — `ListParts`/сравнение размера локального файла).
- Connection Manager и circuit breaker хранят состояние только в памяти процесса — сбрасывается при перезапуске.
- Осиротевший multipart upload на стороне сервера возможен при краше между `CreateMultipartUpload` и записью `multipart_upload_id` в БД.
- Адаптивные тайминги (part size, timeout) — эвристики, не откалиброванные под измеренную скорость канала/RTT.
- Санитайзинг ключей при зеркалировании на диск — базовый (отклонение `..`), нет обработки коллизий «ключ и как файл, и как папка».
- In-memory кэш листинга объектов — без верхней границы/эвикции.
- `GetTextPreview` доверяет MIME-типу, присланному фронтендом — нет серверной проверки.
- Zeroing памяти для ключа шифрования — best-effort (`KeyBox.Clear()`), без криптографических гарантий стирания.

## Инфраструктура и релиз

- **Подписанные инсталляторы** — macOS ad-hoc self-signed, Windows и Linux без подписи вовсе.
- **Реальный AWS S3 в автотестах** — интеграционные тесты идут только против MinIO.
- **Живое тестирование на Windows/Linux** — критерий приёмки для этих ОС — успешная сборка в CI + `go test`, без ручной проверки UI.
- **`.deb`-пакет и `arm64`-сборки Windows/Linux** — сейчас только AppImage (amd64) для Linux и amd64-инсталлятор для Windows.
- **Профилирование RAM/CPU только на macOS arm64** — нет baseline для Windows/Linux. Известный факт: idle RSS (~231 МБ) превышает целевой порог тех-спека (150 МБ) — объяснено архитектурой WKWebView, не утечкой (см. `ARCHITECTURE.md`).
- **Зависимость от `bitnamilegacy/minio`** в интеграционных тестах CI.
- **`internal/profiling`'s pprof-эндпоинт** — не защищён аутентификацией; безопасен, пока `THREEV_PPROF_ADDR` не задан (по умолчанию выключен).
- **Ручная синхронизация версии** между `wails.json`, `frontend/package.json` и git-тегом — частично смягчено `check-version` job в `release.yml`.
- **Полноценный системный трей** — рассмотрен и сознательно отклонён в Этапе 8: `github.com/getlantern/systray` не поддерживается официально Wails v2 (мейнтейнеры прямо заявили «Systray will not be supported in v2», GitHub discussion #4514) и не работает на macOS с Wails v2 из-за конфликта Objective-C линковки. Платформенно-неполное решение (Windows/Linux) с иной UX на macOS признано не стоящим сложности (сторонняя зависимость без официальной поддержки, cgo/GTK-требования к CI, разное поведение по ОС) — не будет реализовываться, пока Wails не даст официальную поддержку.
